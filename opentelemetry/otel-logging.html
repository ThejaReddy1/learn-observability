Logging in
OpenTelemetry

OTel doesn't replace your logging library — it works alongside it. This guide covers log formats, the Log Appender/Bridge pattern, the LogRecord data model, the Logs API and SDK, and how logs connect to the traces already flowing through your system.

Thejasekhar Reddy.G  ·  April 15, 2026

01 — Log FormatsStructured, Unstructured, and Semi-Structured

Before OTel, logs came in three formats — and they still do. Understanding the differences matters because OTel needs to handle all three.

Structured

JSON

Consistent and machine-readable. Every field has a defined key and type. Easy to parse, filter, and query at scale. The format OTel natively works with best.

Unstructured

Free Text

Plain text entries — error stacks, info messages. Human-friendly, hard to parse. Freeform text entries without a fixed schema. Most legacy systems still emit these.

Semi-Structured

Key=Value

Freeform text that includes predictable key-value tokens. Partial structure — human-readable but needs custom parsing. Common in syslog-style outputs.

CLF and ELF

CLF (Common Log Format) is a widely adopted text-based logging standard for web services. Each space-delimited field captures key request details — the client IP, the user identity, the timestamp, the HTTP method and path, the response code, and the payload size.

CLF example log line — space-delimited fields
127.0.0.1 - johndoe [04/Aug/2024:12:34:56 -0400] "GET /index.html HTTP/1.1" 200 1024
#   ^clientIP  ^user identity  ^timestamp                  ^method+path+version  ^code ^bytes

ELF (Extended Log Format) builds on top of CLF by adding structured payloads — often JSON or key-value pairs. It keeps CLF compatibility while adding richer machine-readable data alongside the standard fields.

02 — How OTel Fits InOTel Enriches Existing Logs — It Doesn't Replace Them

OpenTelemetry enriches existing logs and standardizes parsing. It doesn't compete with or create a new logging library — it works alongside your existing logging framework. It uses a Log Appender to add context like the active span and then gets the TraceId and SpanIds, which get attached to the log.

This is the key insight about OTel logging: you keep using logger.info() exactly as you do today. The Log Appender or Bridge sits in between and silently enriches every log record with the trace context that was active when the log was emitted.

OTel Logging Pipeline — Log Appender bridges your existing library to OTel
🔗
Log Appender / Bridge
The component that collects from your existing logging library to OpenTelemetry without changing your code. It hooks into Logback, Log4j, Python logging, JS winston — captures every log record and forwards it to the OTel SDK. Most importantly it adds the active TraceId and SpanId so every log line is connected to the trace that caused it.
🏭
LoggerProvider
Initializes once per application lifecycle. Acts as a factory that creates and manages all the individual loggers used in your app. Configures Resource metadata and Exporters. Part of the Logs Bridge API for library authors.

03 — ComponentsLogger, LogRecord, and LogRecordExporter

A Logger is what your application code actually calls to generate log entries. Your code uses the logger through calls like log.info or log.error to generate log records. Each Logger comes from a LoggerProvider and carries its own InstrumentationScope — metadata identifying which library or component it belongs to.

emit(LogRecord)

The core method to send logs. All emit parameters are optional — LogRecords can include any subset of fields. Timestamp, Severity, Body, Attributes, Context — provide what you have.

isEnabled()

Guard for expensive logging work. Call this before building the log message to avoid unnecessary work. Both methods are thread-safe and safe to call in concurrent environments.

The LogRecordExporter handles Batching, Retry, and Error handling. Pluggable exporters let you swap backends without code changes — from stdout during development to a full Collector pipeline in production.

04 — LogRecordThe Unit of Log Data in OpenTelemetry

A LogRecord is the unit of log data in OpenTelemetry. Every log event — regardless of which library or language generated it — becomes a LogRecord with a consistent structure. LogRecords can include any subset of fields — all emit parameters are optional.

LogRecord — the unit of log data in OpenTelemetry
⏱️
Timestamp
Nanosecond precision. When the event occurred — not when it was processed or exported. Critical for correlating logs with traces that use the same timestamp precision.
🔴
Severity
SeverityNumber 1–24 plus a SeverityText string. Predefined, typed, and present in most records. See the Severity Mapping table in section 05.
📝
Body
The log message. Can be any type — a plain string like "Order processed", or a structured object. Not required to be human-readable.
🔗
TraceId + SpanId
Links this log record to the active trace. Added automatically by the Log Appender when a trace is active. This is what enables log-to-trace correlation in backends like Jaeger and Grafana Tempo.
🏷️
Attributes
Business metadata, additional tags, or labels you can attach freely — order.id, user.id, env. This is where your domain context lives.
🌍
Resource
Describes where this log came from — service.name, deployment.environment, host.name. Shared across all log records from the same process.
📦
InstrumentationScope
The library or module that logged this specific log data. Each log record includes the InstrumentationScope for where it was produced — helping you filter by component in your backend.

05 — Logs Data ModelA Unified Schema for Logs From Any Source

The Logs Data Model specification defines a unified schema for log records from any source — system logs (syslog), third-party (Nginx), or first-party (your custom app). It ensures all of these logs have a consistent representation. The data model ensures raw logs map into the OTel schema and then into export formats like JSON/OTLP/others.

Named Fields

Predefined and typed. Frequent in most records — efficient and type-safe. Ubiquitous means you can see most log records would have these. Examples: Timestamp, TraceId, SeverityNumber.

Map Fields

Arbitrary keys. Less frequent. Flexible and extensible — but watch for duplicate keys. These hold your custom attributes and business metadata.

Severity Mapping Table

SeverityNumber ranges 1–24 map to standard severity categories. If there is a severity mismatch when ingesting from another format, OTel maps to the nearest severity number.

SeverityNumber RangeCategoryMeaning
1 – 4 TRACEA fine-grained debugging event — typically disabled by default
5 – 8 DEBUGA debugging event
9 – 12 INFO An informational event — not an error but more important than debug
13 – 16 WARN A warning event — not an error, but more important than info
17 – 20ERRORAn error event — something went wrong
21 – 24FATALA fatal error — such as application or system crash

06 — Logs APIStandardizing How Logs Are Generated and Emitted

The Logs API standardizes how logs are generated and emitted in OpenTelemetry. It serves library authors, instrumentation libraries, and application developers. The Logs API revolves around four core components.

Logs API — four core components

Initializing the LoggerProvider

LoggerProvider can be initialized as a global default or as a custom instance. Loggers can't be created directly — they come from the provider through getLogger(). Each logger carries its own InstrumentationScope metadata identifying which library or component it belongs to.

Java — initializing LoggerProvider and emitting a LogRecord
// Get the global LoggerProvider (initialized by your SDK setup)
LoggerProvider provider = OpenTelemetry.getGlobalLoggerProvider();

// Get a Logger — name, version, schema, attributes
Logger logger = provider.getLogger(
    "com.example.app",          // name
    "2.1.0",                    // version
    "https://schema.example/",  // schema
    Attributes.of("env", "prod")  // attributes
);

// Guard expensive logging work with isEnabled() first
if (logger.isEnabled(Severity.INFO)) {
    logger.emit(                                  // all params optional
        timestamp  = Instant.now(),
        severity   = Severity.INFO,
        body       = "Order processed",
        attributes = Attributes.of("order.id", 456)
    );
}
Dynamic updates

LoggerProvider updates apply to all loggers — even existing ones

Dynamic updates to LoggerProvider configurations — such as adding a new log processor or changing the log level — automatically apply to all loggers created from it, even the ones that already exist. This means you can change logging behaviour at runtime without restarting your application.

07 — Logs SDKBridging Instrumentation Calls to Actual Telemetry Emission

The OTel Logs SDK bridges instrumentation calls to actual telemetry emission. It implements the Logs API and is required across all supported languages. Most functionality is stable, with some parts still developing.

LoggerProvider is the factory for loggers

It holds shared configuration and can embed a resource describing the service, the environment, or the host. Multiple LoggerProviders allow isolated configurations for different parts of your system.

Lifecycle: Shutdown and ForceFlush

ForceFlush() and shutdown() control the SDK's logging lifecycle — used to immediately push any buffered data. Both return three API signals: Success, Failure, or Timeout.

08 — LogRecordProcessorSimple, Batching, and Custom

LogRecordProcessors sit between the Logger and the LogRecordExporter. They intercept every emitted log record and decide what to do with it before it reaches the export destination.

LogRecordProcessor — three types · arrows go FROM exporter TO backends
⚡
Simple Processor
Sends logs immediately as they are emitted. Good for local testing and debugging. Not suitable for production — synchronous and blocks on every log record.
📦
Batching Processor
Buffers log records and exports them in intervals. Non-blocking — the standard for production. Reduces network overhead by grouping many records into single export calls.
🔧
Custom Processor
Enriches, filters, or routes logs before export. Use this to redact PII, add context, drop noisy logs, or fan out to multiple destinations.

LogRecordExporter Interface

The LogRecordExporter is responsible for sending logs to their destination. It handles protocol-specific transmission. The exporter provides Export(batch) and Shutdown() methods. The export method must be thread-safe and respect configured shutdowns and timeouts. It returns an export result — either success or failure, which guides retry or drop decisions.

Pluggable exporters

Pluggable exporters let you swap backends without code changes — stdout for debugging, Collector for production, a vendor backend for long-term storage. All achieved by changing exporter configuration, not application code.
