OTLP: The Wire Protocol
Powering OpenTelemetry

OTLP is how telemetry data is structured and transmitted over the network. Understanding it is key to understanding how your entire observability pipeline actually works under the hood.

Thejasekhar Reddy.G  ·  April 3, 2026

01 — FoundationsWhat is OTLP?

OTLP is a wire protocol — it defines how telemetry data is structured and transmitted over the network. Think of a wire protocol as a common language that two systems use to exchange data. Just like HTTP defines how web pages are requested and sent, OTLP defines how telemetry data is packaged and delivered.

OTLP does two things. First, it defines the data model — how telemetry data is structured — using Protocol Buffers (protobuf). Second, it defines how that data is transmitted over the wire.

Protocol Buffers

A way to pack data into small, fast, easy-to-send format

Protocol buffers are Google's language-neutral mechanism for serializing structured data. They produce compact binary payloads — much smaller than JSON — which makes them ideal for high-throughput telemetry transmission.

A key design principle: OTLP is signal agnostic. It uses the same approach for all three signals — traces, metrics, and logs. One protocol, one data model, three signal types.

02 — Data ModelThe OTLP Data Structure Model

Every piece of telemetry in OTLP is wrapped in a nested structure that describes where the telemetry is coming from. This context travels with your data all the way to the backend.

OTLP Data Structure Model — Resource → Scope → Signal Data
🏗️
Resource — Outermost container
Metadata describing the source of the telemetry. The Resource is detected automatically or configured manually and contains key-value pairs like service.name, host.name, and deployment.environment. Everything inside this block came from the same resource.
🔭
Scope — Instrumentation context
Within a resource, telemetry is further grouped by the Instrumentation Scope — the specific library name and version that generated the telemetry. This tells you which part of your application emitted the data.
📊
Signal Data — The actual telemetry
Inside the scope sits the actual data: Spans/Trace data, Metrics measurement data, or Logs/Event data. This is the telemetry itself — timestamped, attributed, and ready to be exported.

Resource — Key Concepts

service.name

The most important resource attribute — identifies which service generated the telemetry. Always set this explicitly.

SDK Attributes

Automatically added by the OTel SDK — language, SDK version, telemetry SDK name. You get these for free.

Resource Detectors

Plugins that automatically detect the environment — cloud provider, Kubernetes pod name, host info. Configure once, enriched automatically.

Custom Resources

Set via OTEL_RESOURCE_ATTRIBUTES environment variable. Great for adding deployment environment, team name, or version.

Issue: Namespace Collision

When multiple services or libraries define the same attribute key, the values can collide. Always follow OTel semantic conventions to avoid this.

Serialization

Once telemetry is generated, it's converted into protocol buffers — binaries that are simple and easy to transport. This process is called serialization.

03 — TransportOTLP Transport Protocols

OTLP supports two transport protocols. Both carry the same data — the difference is how that data travels over the network.

OTLP Transport — gRPC vs HTTP
OTLP/gRPC

Streaming-based · Default

Uses the gRPC framework with Protobuf Binary Encoding over HTTP/2. gRPC is a streaming-based approach — it keeps a persistent connection open and streams telemetry continuously. This is the default and preferred transport for most production setups.

Uses a simple endpoint with gRPC service methods like ExportTraceServiceRequest. Supported by default on port 4317.

default: port 4317
OTLP/HTTP

Request-based · Simpler alternative

Uses HTTP POST requests with either Protobuf Binary or JSON encoding over HTTP/1.1 or HTTP/2. HTTP is closer to a traditional request-based approach — each export is a separate POST request.

Exports by appending the full URL path for each signal type, then transmitted. Need to specify compression separately.

default: port 4318

OTLP/HTTP Endpoint Paths

When using OTLP/HTTP, telemetry is sent to signal-specific endpoints via HTTP POST:

🔗
/v1/traces
Trace and span data. Content-Type: application/x-protobuf for binary, or application/json for JSON encoding.
📈
/v1/metrics
Metrics measurement data. Same content-type options as traces.
📋
/v1/logs
Log records and event data. Each signal has its own endpoint — the Collector receives them independently and routes them through their respective pipelines.

gRPC vs HTTP — When to Use Which

Use gRPC when...

You need high throughput and low overhead. Both sides support HTTP/2. You're in a controlled internal network. This is the default for most Collector configs.

Use HTTP when...

You're sending telemetry through firewalls or load balancers that don't support HTTP/2. You need JSON payloads for debugging. You're integrating with systems that don't support gRPC.

Our take

Start with gRPC — it's the default and works well in most environments. Switch to OTLP/HTTP only if you hit network or firewall issues. The JSON encoding option in OTLP/HTTP is particularly useful when you need to inspect payloads during debugging.

04 — Putting It TogetherOTLP in Your Pipeline

In practice, OTLP is the glue between every component in your observability pipeline. Your app's OTel SDK serializes telemetry into protobuf and sends it via OTLP to the Collector. The Collector receives it, processes it, and re-exports it via OTLP to your backends. Every hop uses the same protocol — same data model, same wire format.

# Collector config: receive OTLP, export OTLP
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

exporters:
  otlp:
    endpoint: https://your-backend:4317
    compression: gzip
💡 Compression

Both gRPC and HTTP transports support gzip compression. Always enable it in production — it typically reduces payload size by 70-90% for telemetry data, dramatically reducing egress costs.
