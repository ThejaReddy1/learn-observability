What is OpenTelemetry?

A clear, no-jargon explanation of what OTel is, the problem it solves, and why the entire observability industry is converging on it — plus our honest take on where it shines and where it still has rough edges.

Thejasekhar Reddy.G ·  April 2, 2026

01 — The Problem Before OTelObservability Was a Tower of Babel

Before OpenTelemetry, every observability vendor had their own agent, their own data format, their own SDK. You wanted to send metrics to Prometheus? One SDK. Traces to Jaeger? Another SDK. Logs to Elasticsearch? Yet another.

Your application code was littered with vendor-specific instrumentation. Switching tools meant rewriting your instrumentation from scratch. Teams got locked in — not because the products were great, but because leaving was too painful.

"Observability had a standardization problem. Every vendor spoke a different language, and your application had to learn all of them."

In 2019, two overlapping open source projects — OpenCensus and OpenTracing — merged to form OpenTelemetry. The goal was simple and ambitious: create a single, vendor-neutral standard for generating, collecting, and exporting telemetry data.

02 — What OTel Actually IsA Standard, Not a Backend

The most important thing to understand about OpenTelemetry: it is not a database, not a dashboard, and not a monitoring backend. It's a standard — a set of APIs, SDKs, and tools for how telemetry data should be generated and transported.

Think of it like HTTP. HTTP doesn't store your web pages — it defines how they're requested and sent. OpenTelemetry doesn't store your metrics — it defines how they're generated and exported. You still need a backend (Prometheus, Jaeger, Grafana, Datadog, etc.) to store and visualize the data.

OTel in one sentence

Instrument once. Export anywhere.

Write your instrumentation using OTel's open APIs and SDKs. Then route your telemetry to any backend that supports the OTel protocol — without changing your application code.

03 — The Three PillarsMetrics, Logs, and Traces

OpenTelemetry covers the three core types of telemetry data — often called the three pillars of observability:

The Three Pillars of Observability
📈
Metrics
Numerical measurements over time. CPU usage, request count, error rate, latency percentiles. Good for understanding trends and triggering alerts. OTel metrics are compatible with Prometheus.
📋
Logs
Timestamped records of discrete events. What happened, when, and on which service. OTel connects logs to the traces that caused them — giving you context you wouldn't have with raw log lines alone.
🔗
Traces
The path a request takes through your system — across services, databases, queues. A trace is made up of spans, and each span represents one operation. Traces are the only way to truly understand distributed system behavior.

What makes OTel powerful is that these three aren't siloed. A trace can reference the logs generated during that request. Metrics can be derived from span data. The context flows between them — which is what turns raw data into actual insight.

04 — The Key ComponentsWhat You're Actually Working With

The API

A set of language-specific interfaces for instrumenting your code. You call the OTel API to create spans, record metrics, and emit log records. The API is intentionally minimal — it's designed to be embedded in libraries without pulling in heavy dependencies.

The SDK

The implementation behind the API. The SDK handles sampling, batching, and exporting your telemetry. You configure the SDK once at startup — telling it where to send data — and your instrumented code just works.

The Collector

A standalone service that receives telemetry from your apps, processes it, and exports it to one or more backends. Most production setups run a Collector — it decouples your apps from your backends and lets you do filtering, sampling, and transformation in one place. We wrote a full guide on the Collector if you want to go deeper.

OTLP

The OpenTelemetry Protocol — the wire format used to transmit telemetry data. If something says it "supports OTLP", it means it can receive (or send) OTel data natively. Most modern observability backends support OTLP today.

05 — Our TakeWhere OTel Is Great — and Where It's Still Rough

Our honest opinion

OpenTelemetry is the right foundation. But that doesn't mean it's painless — especially if you're just getting started.

Where OTel genuinely shines:

Vendor freedom

Instrument once, send to any backend. Switching from one tool to another no longer means rewriting all your instrumentation. This alone is a massive unlock.

Auto-instrumentation

For many languages and frameworks, OTel can instrument your code with zero changes. Just add the agent and you get traces and metrics out of the box.

Semantic conventions

OTel defines standard attribute names across languages and frameworks. A database span looks the same whether it came from Java or Go — making dashboards and queries portable.

Growing ecosystem

Every major framework, cloud, and observability vendor now supports OTel. The momentum is undeniable — and it's only accelerating.

Where it still has rough edges:

⚠️
Configuration complexity
The OTel Collector config can get complicated fast. YAML pipelines with receivers, processors, and exporters are powerful but not beginner-friendly. Budget time to learn it properly.
⚠️
Agent management is unsolved
OTel tells you how to collect and export data — but not how to manage the agents doing it at scale. OpAMP is an emerging open protocol that addresses exactly this gap.
⚠️
Logs are still maturing
OTel metrics and traces are stable and production-ready. The logs specification has been slower to stabilize — though it's getting much better.

06 — The Bottom LineShould You Adopt OTel?

If you're starting a new project: yes, without question. Instrument with OTel from day one. You'll thank yourself when you need to switch backends, onboard new tools, or hand your observability setup to a new team member.

If you're migrating an existing system: be strategic. You don't need to rip everything out at once. Start with new services, instrument them with OTel, and migrate existing ones gradually.

The observability industry is converging on OpenTelemetry. The question isn't whether to adopt it — it's when and how fast.
