OTel Instrumentation Part 2:
Libraries and Auto-Instrumentation

Maximum observability with minimum effort — but with important limitations to understand. This guide covers instrumentation libraries and the critical API-only constraint, the three techniques behind zero-code instrumentation, the Java agent, Python zero-code setup, and how to choose the right strategy.

Thejasekhar Reddy.G  ·  April 4, 2026

01 - Instrumentation LibrariesPre-built Packages That Hook Into Your Libraries

OTel Instrumentation Libraries are pre-built packages that automatically add tracing, metrics, and logs to popular libraries and frameworks. They use language-specific techniques like Python's monkey patching to wrap existing functions and capture telemetry seamlessly. No code changes required on those sections of code.

One line like RequestsInstrumentor().instrument() gives full tracing coverage for every HTTP call your application makes. The instrumentation library handles starting and ending spans, adding semantic attributes, and propagating context in and out. How-to-collect is handled by library-based instrumentation.

Instrumentation Library — depends on OTel API only · SDK + exporter is your responsibility
📦
Libraries depend on OTel API ONLY — not the SDK
This is the critical architectural fact. Third-party libraries and frameworks that add instrumentation depend only on the OTel API. The API calls are no-ops with minimal overhead when no SDK is configured. When the SDK is installed and configured by the application owner, the API calls start producing real telemetry. Libraries are designed this way so they remain fully usable in applications that don't want OpenTelemetry at all.
🛠️
Exporting is YOUR responsibility — the library never touches the SDK
Exporting telemetry data is not done by instrumentation libraries. You must configure the SDK separately — TracerProvider, BatchSpanProcessor, and OTLPExporter. Without that setup, the library creates spans but they go nowhere. The library only uses the OTel API. The SDK and its configuration are your responsibility as the application owner.
Examples

opentelemetry-instrumentation-requests for HTTP calls, opentelemetry-instrumentation-django for Django apps, opentelemetry-instrumentation-sqlalchemy for database queries. Each hooks into the library's behaviour, starts and ends spans, and adds semantic attributes automatically.

02 - Zero-Code InstrumentationAuto-Inject Telemetry With No Code Changes

Auto instrumentation automatically injects instrumentation to generate telemetry data with no code changes. It injects OpenTelemetry APIs and the OpenTelemetry SDK into your application at startup. There is one important limitation: no custom business logic instrumentation can be done using zero-code instrumentation. For that, you still need manual instrumentation.

Three Techniques Behind Zero-Code Instrumentation
①
Monkey Patching — Python, JavaScript
Dynamically replaces or patches functions at runtime. Telemetry code is injected into existing methods — so every HTTP call, DB query, or cache operation is automatically tracked because the function itself has been replaced at the moment of import.
②
Bytecode Manipulation / Bytecode Injection — Java, .NET
Agents dynamically modify the compiled bytecode, inserting telemetry calls before or after method execution. Your source code is untouched — the modification happens to the compiled class files at JVM or CLR load time. This is how the Java agent works.
③
OTel eBPF Instrumentation (OBI) — kernel level
The instrumentation works at kernel level, observing system calls and network activity. It is ideal for containerised or polyglot environments where you cannot install a per-language agent. Works by observing kernel-level events rather than patching application code.

Configuration Options

We can configure or tweak auto instrumentation using three options. Environment Variables — the most common approach, works everywhere including containers and Kubernetes. System Properties — especially in Java or .NET environments, passed as JVM flags. Startup Arguments — CLI flags passed to the instrumentation command. Common things you configure: service name, exporter (OTLP or Console), propagator (W3C, B3), and resource attributes.

03 - Java AgentRunning with the Java Agent

The OpenTelemetry Java Agent injects instrumentation into supported frameworks and libraries at runtime by attaching to the JVM at startup using the -javaagent flag. Running in the basic form alone is not recommended — you need to configure it properly with System Properties or environment variables.

Java Agent — attaches at JVM startup, injects bytecode, zero code changes in your app

Customising with System Properties

These flags are passed as JVM system properties. The default batch size of BatchSpanProcessor is 512.

Java Agent Extensions

Java Agent Extensions allow you to enhance or customise Java Agent behaviour without modifying the agent itself or creating a separate build. Extensions open the door to four major capabilities.

1
Add custom span processors, exporters, samplers and propagators
Plug in your own components alongside the built-in ones without forking the agent.
2
Inject new instrumentation modules
Add instrumentation for a library the agent does not support out of the box.
3
Modify or filter span attributes (PII data) before it leaves the system
Redact sensitive data even before it reaches the Collector or backend.
4
Override or disable existing instrumentation
Turn off instrumentation for specific libraries that generate too much noise.
Java Agent Extensions — build and run
# Build extension — produces a jar in build/libs/
./gradlew build

# Run with a single extension
java -javaagent:opentelemetry-javaagent.jar \
     -Dotel.javaagent.extensions=/path/to/extension.jar \
     -jar myapp.jar

# Multiple extensions — comma separated JARs
java -javaagent:opentelemetry-javaagent.jar \
     -Dotel.javaagent.extensions=/ext1.jar,/ext2.jar \
     -jar myapp.jar

04 - Python Zero-CodeZero-Code Instrumentation in Python

OTel behaves like an agent by applying monkey patches to supported libraries. These patches modify library functions at runtime so every request, query, or call made by your app is automatically tracked. Flask, Django, requests, SQLAlchemy, Redis — all automatically instrumented. No manual instrumentation required.

Python Zero-Code — bootstrap scans your env, opentelemetry-instrument wraps your app

Advanced Configuration

🔗
URL Exclusions
export OTEL_PYTHON_EXCLUDE_URLS="/health,/metrics" — exclude health check and metrics endpoints from tracing. Keeps your traces clean.
📋
Request Attributes (Django)
export OTEL_PYTHON_DJANGO_TRACED_REQUEST_ATTRS="Path_info,content_type" — specify which request attributes to capture on Django spans.
🔄
Log Correlation
Set OTEL_PYTHON_LOG_CORRELATION=true and OTEL_PYTHON_LOG_FORMAT="%(msg)s [trace_id=%(otelTraceID)s]" to inject the TraceId into every log line — so you can jump from a log directly to its trace in the backend.
🚫
Disabling Instrumentation
export OTEL_PYTHON_DISABLE_INSTRUMENTATIONS=redis,grpc,kafka — disable specific library instrumentations. export OTEL_SDK_DISABLED=true disables the entire SDK.

05 - StrategyChoosing the Right Instrumentation

The right choice depends on how much data control you want. In real-world scenarios, a layered instrumentation strategy is most effective.

🤖

Auto Only

Fast to adopt. Limited to what libraries can see. No business context — framework-level spans only.

✍️

Manual Only

Maximum data control. Full observability where it matters. Requires more code and deliberate effort.

⚡

Layered (Recommended)

Auto instrumentation first for coverage. Layer in manual for business logic. Most effective in real-world scenarios.

"Start with auto instrumentation. Then go for manual or code-based instrumentation for business logic. Based on how much data control you want — then go accordingly. In real world scenarios, a layered instrumentation strategy is most effective."
