OTel Instrumentation Part 1:
SDK, Tracing API and Sampling

Instrumentation is the process of adding code or tooling that generates telemetry to observe application behaviour. This guide covers the two approaches, how the SDK and Tracing API work together, span lifecycle from start to end, processors, exporters, and sampling.

Thejasekhar Reddy.G  ·  April 4, 2026

01 — ApproachesThree Ways to Add Instrumentation

OpenTelemetry gives you three instrumentation approaches, and understanding the difference is the foundation for everything else.

✍️
Manual Instrumentation
You write code using the OTel Tracing API to explicitly create spans, set attributes, record events, and handle exceptions. Full control — gives you the richest data especially for business logic — but requires deliberate effort for every operation you want to trace.
🤖
Automatic Instrumentation
Auto instrumentation automatically injects instrumentation to generate telemetry data with no code changes. Uses language-specific techniques like monkey patching to wrap existing functions and capture telemetry seamlessly.
📦
Library-based Instrumentation
Pre-built packages that automatically add tracing and metrics to popular libraries. Zero-touch observability — one line like RequestsInstrumentor().instrument() gives full tracing coverage. Follows standardised data via semantic conventions.

02 — Tracing API WorkflowWhat Happens When You Create a Span

Every span goes through this exact journey per the OTel specification. Understanding the order matters because sampling happens during span creation, not before or after it.

Tracing API Workflow — per OTel specification

The key insight: the Sampler's ShouldSample is called by the Tracer during span creation. If the sampler decides to drop a span, the span becomes a NonRecordingSpan and never reaches the SpanProcessor or SpanExporter. The SpanId is generated independently of the sampling decision — so even dropped spans have a valid ID, useful for logs and exception handling.

03 — Core ComponentsTracerProvider, Tracer, Span, SpanExporter

SDK Components — TracerProvider owns Sampler, SpanProcessor, SpanLimits
🏭
TracerProvider — Entry point that owns all configuration
The factory that creates and manages Tracers. Configuration — SpanProcessors, Sampler, SpanLimits — is owned by the TracerProvider. Set up once at startup. Without one configured, OTel falls back to a no-op provider where telemetry is generated in memory but never recorded or exported.
📡
Tracer — Creates Spans, handles in-process context propagation
What you use in your code to create spans. All Spans MUST be created via a Tracer — there is no other API for creating a Span. Get a named Tracer per module using get_tracer(__name__). The Tracer controls access to the execution context and manages span creation and span lifecycles.
📦
Span — Single timed operation, immutable after end()
The fundamental unit of a trace. Once a span has completed, it is immutable and can no longer be modified. Only the first call to end() modifies the span — implementations are free to ignore further calls. Always end spans properly using context managers so they end even when exceptions occur.
📤
SpanExporter — Encodes and sends span data to a telemetry backend
Picks up the spans, encodes them into the format the backend expects, and delivers them. Use OTLPExporter with BatchSpanProcessor in production. Use ConsoleSpanExporter with SimpleSpanProcessor during local development.

03b — In ActionSDK Setup and Span Lifecycle in Code

Here is what setting up the SDK and creating your first span looks like in Python. Pay close attention to where the span starts, where work happens, and where span.end() is called — and critically, how error recording works.

Python — SDK setup, span lifecycle, correct error handling
# pip install opentelemetry-sdk opentelemetry-exporter-otlp
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

# 1. Configure TracerProvider once at startup
#    TracerProvider owns: Sampler, SpanProcessor, SpanLimits
provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="http://localhost:4317"))
)
trace.set_tracer_provider(provider)

# 2. Get a named Tracer — name identifies your component
#    All spans MUST be created via a Tracer
tracer = trace.get_tracer("checkout-service", version="1.0.0")

# 3. start_as_current_span: creates span AND sets it active in context
#    Context manager calls span.end() automatically when block exits
#    Even if an exception is raised — span always ends cleanly
with tracer.start_as_current_span("process-checkout") as span:
    # Prefer setting attributes at creation — available to samplers
    span.set_attribute("user.id", user_id)
    span.set_attribute("order.id", order_id)

    try:
        result = process_order(order_id)
        # Set OK explicitly to confirm success
        span.set_status(trace.StatusCode.OK)

    except Exception as e:
        # record_exception records the exception as a span event
        span.record_exception(e)
        # IMPORTANT: record_exception does NOT set status automatically
        # You MUST call set_status separately
        span.set_status(trace.StatusCode.ERROR, str(e))
        raise
# span.end() was called by context manager — span is now immutable
Critical: record_exception does NOT set status

You must call set_status(ERROR) separately

span.record_exception(e) records the exception as a span event with a timestamp. It does not automatically set the span status to ERROR. You must call span.set_status(StatusCode.ERROR, description) separately — otherwise your span appears green in your backend even though it failed.

No-op fallback — what happens without a TracerProvider
# Without set_tracer_provider() — falls back to no-op
tracer = trace.get_tracer("my-service")

with tracer.start_as_current_span("my-span") as span:
    # span is a NonRecordingSpan
    # telemetry generated in memory but never exported
    # your code still works — nothing breaks
    do_work()

# Intentional for library authors: libraries can use OTel API safely
# even when the application has not configured any observability
Language neutral — same span lifecycle, different syntax for ending spans
# Python — context manager ends span automatically
with tracer.start_as_current_span("my-span") as span:
    do_work()
# span.end() called here automatically

// Go — defer guarantees end even on panic
ctx, span := tracer.Start(ctx, "my-span")
defer span.End()
doWork(ctx)
// span.End() called when function returns

// Java — startActiveSpan handles end via callback return
tracer.startActiveSpan("my-span", span -> {
    doWork();
    span.end();
    return null;
});

// Node.js — must call end() explicitly inside callback
tracer.startActiveSpan("my-span", (span) => {
    doWork();
    span.end(); // must call explicitly
});

04 — Tracing APICore API Features Reference

The OTel Tracing API is consistent across all languages. Note the distinction between start_span() and start_as_current_span() — the former does NOT set the span as active in context by default.

05 — Processors and ExportersHow Spans Move From SDK to Backend

The SpanProcessor sits between the TracerProvider and the SpanExporter. It hooks into onStart() when the span begins and onEnd() when span.end() is called. It only runs if the span is recording — dropped spans never reach the processor.

SpanProcessor — sits between TracerProvider and SpanExporter

SimpleSpanProcessor

Passes each finished span directly to the exporter as soon as span.end() is called. Synchronous — blocks the calling thread. Good for local testing. Not suitable for production.

BatchSpanProcessor

Creates batches of finished spans and passes them to the exporter asynchronously. Non-blocking — the default for production. Default maxQueueSize: 2048. Default scheduledDelay: 5000ms.

The SpanExporter has three key behaviours. export() sends a batch of spans to the destination. shutdown() stops exporting and cleans up resources — call before exit to avoid data loss. forceFlush() immediately pushes all pending spans regardless of batching.

06 — SamplingDeciding Which Spans Get Kept

Sampling is the mechanism that decides which spans are kept and which are dropped. Sampling decisions are propagated across services using TraceFlags. The decision happens during span creation and influences not just the current span — child spans created with ParentBasedSampler will also be dropped if the parent is not sampled.

Sampling — ShouldSample called during span creation
AlwaysOnSampler
Dev - Low volume - Debugging

Records all spans. Use in development or low-volume production services where you want full visibility.

AlwaysOffSampler
Disable tracing without removing code

Records no spans. Useful when you want to disable tracing without removing instrumentation from your codebase.

TraceIdRatioBased
Production - High volume

Records a percentage of spans. Head-based sampling. Good for production workloads at scale — e.g. 20% sampling with OTEL_TRACES_SAMPLER_ARG=0.2.

ParentBasedSampler
Distributed systems consistency

Follows the sampling decision of the parent span. Ensures entire traces are either fully sampled or fully dropped — no partial traces in your backend.

Configure via environment variables for Auto-Instrumentation

OTEL_TRACES_SAMPLER=parentbased_traceidratio and OTEL_TRACES_SAMPLER_ARG=0.2 for 20% sampling. Default is parentbased_always_on.

07 — Span AttributesAssigning Metadata for Better Visibility

Span attributes are key-value pairs that contain metadata you can use to annotate a span. Prefer adding attributes at span creation — attributes set after creation are not available to SDK samplers.

Semantic Attributes

Predefined standard keys — http.method, db.system, http.status_code. Set automatically by instrumentation libraries. Ensure consistency across services and backends.

Custom Attributes

Your domain context — user.id, order.value, tenant.name. Not automatic. You add them manually. These are what make traces genuinely useful during a real incident.

08 — Best PracticesTracing Best Practices

✅
Always end your spans — use context managers or defer
Only the first call to end() takes effect. Use Python context managers, Go defer span.End(), or Java try-with-resources. A span that never ends leaks resources and appears missing in your backend.
✅
Call set_status(ERROR) after record_exception
record_exception() does not automatically set the span status. Always call set_status(StatusCode.ERROR) separately if the span tracks a failed operation — otherwise your span appears as a success.
✅
Set up one TracerProvider globally or via Dependency Injection
One provider, configured at startup, shared across your entire application. Never create multiple TracerProviders unless you need explicitly isolated configurations.
✅
Get named Tracers per module or service
Use get_tracer(__name__) in each module so you know exactly which component generated each span. The name becomes the InstrumentationScope in your backend.
✅
Use context propagation for distributed tracing
Always inject context into outgoing requests and extract it from incoming ones. Without this, traces break at service boundaries and you lose the full picture.
