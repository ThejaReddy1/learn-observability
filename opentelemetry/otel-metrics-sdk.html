01 — Metric InstrumentsThe Seven Instruments and When to Use Each

Instruments are defined by name, type, and optional details such as units, description, and advisory hints. The instrument type determines the behavior of the data — how it's recorded, aggregated, and what it represents. Choosing the right instrument is based on how you want the data to appear and behave in your backend.

Synchronous Instruments

Records metric measurements inline with application logic and carry execution context. Context-aware — helps in correlating with traces and logs. Called directly in your code at the moment of measurement.

Asynchronous Instruments

Use callbacks triggered during collection, so they report on demand without context. Cannot access execution context. Ideal for metrics such as CPU time that are better read periodically rather than inline.

Counter

📈
Counter — Synchronous, Monotonic, Context association supported
Records counts. Values only go up. Use when you need to count occurrences over time — http.requests.total, errors.count, bytes.sent. Aggregation: Sum (monotonic cumulative).
🔄
Async Counter (ObservableCounter) — No context association, Cumulative metrics
Reports cumulative, monotonic values via callbacks only when observed. Ideal for metrics such as CPU time — gives total values. Use when reading the metric directly from the system rather than counting in code.

Gauge

🌡️
Gauge — Synchronous, Non-additive, Context association supported
Used when a current state of something needs to be measured. The values can go up and down over time. Non-additive values — the current value is all that matters. Context association supported. Examples: CPU usage, Memory usage, Queue Length, Decibel levels, Number of active users.
📷
Async Gauge (ObservableGauge) — No context association, snapshot metrics
Invokes a callback at collection time to get the current value on demand. No context association. Snapshot metrics. Useful for: disk space remaining, battery level, thread count, temperature/humidity — reported by a callback at time of collection.

UpDown Counter

↕️
UpDownCounter — Synchronous, Non-monotonic, Context association supported
Allows both increments and decrements, making it ideal for tracking dynamic counts. Non-monotonic — values can go up and down. Reports value directly. Use cases: active connections, number of items in cart, concurrent requests, active users.
🔁
Async UpDownCounter — Callback-based, Additive values
Callback-based. Additive values. Picked up on observation. No context association. Use cases: number of running processes, open file descriptors, active VMs, connected clients — reported by a callback at timeseries.

Histogram

📊
Histogram — Synchronous, Non-additive, Non-monotonic — ideal for performance analysis
A Histogram is a synchronous instrument capturing arbitrary measurements to produce statistical summaries such as percentiles. Records arbitrary values. Records distributions. Monotonicity not required. Summaries and percentiles. Records API response time percentiles (p99, p95). Example: http.server.duration tracking request duration percentiles in a web server.

Choosing the right instrument

It is based on how you want the data to appear and behave in your backend

Count occurrences over time → Counter. Values can go up and down → UpDownCounter. Tracking current states or live readings → Gauge. Need statistical summaries like percentiles → Histogram. Reading from system callbacks → use the Async variant of any instrument.

02 — Metrics API & SDKThree Components That Work Together

The Metrics API defines the interfaces for creating and reporting metrics. The SDK actually implements these interfaces to collect and export the data. The API has 3 components — MeterProvider, Meter, and Instrument — that work together to capture measurements.

🏭
MeterProvider — Central Configuration Point
Serves as the main entry point of the metrics providers. Think of this as a factory — that is entry point to get a tool (Meter). Offers getMeter() function, accepting name, version, schema URL, and attributes (optional, to enhance context). You can use Default metric providers or use custom ones for advanced scenarios.
🔧
Meter — Creates Instruments, sole responsibility is on Instrument creation
Think of it like a tool. It is responsible for creating Instruments of various kinds like counters, gauges etc. Its sole responsibility is on Instrument creation and not configuration. It creates instruments by using dedicated functions: createCounter, createGauge, createHistogram, createAsyncCounter, createAsyncGauge, createUpDownCounter (both Sync and Async).
📦
Instrument — Defines what to measure
The instrument itself — like those counters, gauges, histograms — are the ones that would actually report the measurements. An Instrument represents a metric you record, defined by its properties: Name, Kind, Unit, Description. The properties that the data type play a key role in Register and Retrieve the data.

Configuration of the instrument is delegated to the MeterProvider. Each instrument is linked to the Meter that created it, ensuring clear ownership — because each meter is clearly linked to its meter provider, developers always know where an instrument belongs and who manages its configuration.

Creating a Meter and Counter — Python
# Get the MeterProvider (initialized by your SDK setup)
from opentelemetry import metrics

# Get a Meter — name is the instrumentation scope
# avoid empty and vague names
meter = metrics.get_meter("http-client", version="1.2.0")

# Create a Counter instrument
request_counter = meter.create_counter(
    name="http.requests.total",
    unit="1",
    description="Total number of HTTP requests"
)

# Record a measurement — add() for Counter
request_counter.add(1, attributes={"http.method": "GET", "http.status_code": 200})

# Create a Histogram instrument
duration_histogram = meter.create_histogram(
    name="http.server.duration",
    unit="ms",
    description="HTTP request durations"
)
duration_histogram.record(45.2, attributes={"http.route": "/api/checkout"})
Note

Avoid empty and vague names when creating a Meter. The Meter name is the instrumentation scope identifier — it tells you which component or library the metrics came from. Use a specific, meaningful name like http-client or com.example.checkout.

03 — Metrics PipelineFrom Instrument to Backend

The SDK provides the concrete behavior behind these abstractions, enabling users to record telemetry and then send those telemetry to backend.

Metrics Pipeline — MeterProvider → Meter → Instrument → Measurement → View → Reader → Exporter
📋
Measurement
A single data point with value and attributes — the data points that are being created by Instruments.
🔍
View (optional)
Configuration layer that reshapes how metrics are collected, aggregated, and exported. Configures aggregation and filtering of metric data. This is the configuration layer — how raw data points become summaries.
📡
MetricReader
Collects and aggregates measurements at defined intervals. The Reader decides how numbers are combined (aggregated) and over what time (temporality). Always works as a partner with MetricExporter.
📤
MetricExporter
Encodes and sends aggregated metric data to a telemetry backend. A MetricExporter is a small plugin that sends your metrics out. They get already-summarized metrics from the reader. Exporter should stay simple — just pack the data and send it. If an exporter can't handle the data's format or timing, it should log an error (you can fix it by changing reader settings).

Exemplars connect metric measurements to trace context. They are sample data points associated with specific time series in the metrics, for reproducibility.

04 — Metric ExportersPush, Pull, and the Lifecycle

Push vs Pull MetricExporter — two export modes

Push-based MetricExporter

Sends metrics out of its own. Always works with a paired MetricReader that feeds it data. It can send on a schedule you choose (e.g. every 1 min) — usually by using a periodic MetricReader. It can also send immediately when there's a serious error. You can use many exporters at once (push, pull, or both), each with its own reader. Each exporter can run on its own schedule.

Pull MetricExporter

Exposes metrics and waits to be scraped (e.g. Prometheus). Data isn't pushed — it's served on request, the exporter is passive. Because the scraper decides timing there is no own schedule and ForceFlush doesn't apply.

SDK Lifecycle — Shutdown and ForceFlush

When application shuts down, the lifecycle perspective of the metrics pipeline calls the Shutdown API. ShutdownAPI is a termination API for providers, invoked exactly once to perform cleanup by calling shutdown on Registered MetricReaders and exporters. The SDKs may implement it as a Blocking or Asynchronous call with a timeout. It should report whether it succeeded, failed, or timed out.

During Shutdown

If new meters are requested, no-op meters are provided. Before shutdown we often perform flush operation by using ForceFlush. Pull exporters stop exposing the scrape endpoint — after shutdown, scrapers won't get data.

ForceFlush

An API that prompts all push-based exporters to immediately collect and force flush on each MetricReader. ForceFlush doesn't apply to pull exporters. Both return Success, Failure, or Timeout.

05 — ExemplarsConnecting Metric Data to Trace Data

Exemplars carry extra context like TraceID or SpanIDs. Exemplars are sample data points associated with specific time series in the metrics. They capture the measurement context of specific measurements when your application records a metric value.

Purpose: enables correlation of metric data with trace data, giving you deeper observability — connecting what happened in metrics to why it happened in traces.

Exemplars — connecting metric measurements to trace context
Value
Simply the raw measurement — like the number passed into the counter.add() or histogram.record().
Timestamp
When the measurement was made.
Filtered Attributes
Any labels excluded or simplified via Views.
TraceContext — trace_id + span_id
The active trace at the time of measurement. The specific span where the measurement occurred. Set up exemplars once with an active trace, and the metric-to-trace linking happens automatically. API and SDKs for metrics and traces are aware of the creation of metric and trace, and they are linked automatically. The visualization layer handles most of the linking between metrics, traces and logs.
