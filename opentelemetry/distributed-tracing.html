Distributed Tracing
with OpenTelemetry

When a request touches six services and something breaks, where do you start? Distributed tracing answers that question. Here's how it works — from spans and context all the way to propagation across service boundaries.

Thejasekhar Reddy.G  ·  April 5, 2026

01 — FoundationsWhat is a Distributed Trace?

A distributed trace shows the full journey of a request across all your services and systems. It gives you end-to-end visibility — the complete picture of what happened, in which service, in what order, and how long each step took.

The fundamental unit of a trace is a span. A span represents a single, named unit of work — one HTTP call, one database query, one function execution. Every span in a trace shares the same TraceId. That shared ID is what links them all together into one complete picture.

"Think of a trace as a tree of spans — the root span starts the request, and every downstream call creates a child span hanging off it."

Distributed Trace — Request flowing across services as a waterfall of spans

The root span has no parent — its ParentSpanId is null. Every other span points back to its parent. This parent-child relationship is how your backend reconstructs the waterfall view you see in tools like Jaeger or Tempo.

02 — Inside a SpanWhat Every Span Carries

A span is more than just a name and a duration. It carries a rich set of fields that together tell the full story of one operation.

Inside a Span — every field and what it carries
🏷️
Name
A meaningful description of the operation. The convention is {verb} {object} — for example GET /users, db.query orders, or send email. This is what you'll see in your tracing backend — make it descriptive.
🔑
SpanId
Unique within a trace. 8 bytes (64 bits). Forms the parent-child relationship — every non-root span has a Parent SpanId pointing to its parent.
👨‍👧
Parent SpanId
Null if this is the root span. Otherwise, points to the parent span's ID — this is how the Collector and backends reconstruct the relationship between spans into a tree.
⏱️
Start and End Timestamps
Recorded with nanosecond accuracy in epoch time. Duration is end_time − start_time. Parent spans usually wrap their children — the parent's duration covers the time of all its child spans combined.
📦
SpanContext (metadata)
An immutable object containing the TraceId, SpanId, TraceFlags, and TraceState. SpanContext is the serialized part sent with Distributed Context and Baggage with every outgoing request — it's how trace identity crosses service boundaries. More on this in section 03.
📋
Attributes (descriptive details)
Key-value pairs of metadata attached to the span. They give your backend something to filter and search on. Two types: semantic attributes (standardized keys like http.method or db.system) and custom attributes (your business context — user.id, order.value, tenant.name). Semantic attributes come automatically from instrumentation libraries. Custom ones you add manually.
📌
Span Events
Timestamped annotations within a span — they mark a specific moment in time during the span's execution. Events have no start or end of their own, just a timestamp. All exceptions must be recorded as span events. When you call add_event() without a timestamp, the SDK assigns the current time automatically.
🔗
Span Links
Relationships between spans that are related but not parent-child. Used in async scenarios — a Kafka consumer starting a new trace can link back to the producer's span, even though they're in completely different traces. Span links can cross trace boundaries.
✅
Status
UNSET (default) — span completed without any status being set. OK — explicitly confirms success. ERROR — something went wrong. Set ERROR when an exception occurs so your backend can surface it properly.

03 — Span ContextThe Identity That Travels With Every Request

SpanContext is the immutable part of a span that gets serialized and sent across service boundaries. While the rest of a span stays in your backend, SpanContext travels with every outgoing request so the receiving service can continue the same trace.

SpanContext — The Immutable Identity Passed Between Services
🆔
TraceId — 16 bytes · 128-bit
Unique per trace. Represented as a 32-character hex string. Every single span in a trace shares this same TraceId — it's the thread that connects them all.
🔢
SpanId — 8 bytes · 64-bit
Unique within the trace. Combined with ParentSpanId, this forms the parent-child tree that your backend uses to render the waterfall view.
🚩
TraceFlags — 8-bit · mandatory
Always present in every request. Controls sampling — 00 means not sampled, 01 means sampled. Unlike TraceState, this field is never optional.
🏪
TraceState — optional · max 256 chars
Carries vendor-specific metadata alongside the trace. Key-value pairs using the ot:=value reserved prefix. Useful for passing vendor flags or configuration between systems.

04 — Span KindClassifying the Role of Each Span

Span Kind classifies what role a span plays in the distributed system. It tells your backend and instrumentation libraries whether a span is initiating a request or responding to one — which matters for dependency mapping and latency attribution.

CLIENT
outgoing →
Outgoing HTTP calls, DB queries. The span making the request.
SERVER
← incoming
Incoming HTTP requests. The span receiving and handling the request.
PRODUCER
outgoing →
Publishing a message to a queue or topic. Deferred execution.
CONSUMER
← incoming
Consuming async messages from a queue. Often uses Span Links.
INTERNAL
local ↔
Local operations, business logic, transformations. The default.
HTTP pattern

Every outgoing HTTP call from Service A becomes a CLIENT span. Every incoming HTTP call received by Service B becomes a SERVER span. Together they form the cross-service link in your trace waterfall.

HTTP Instrumentation Pattern

CLIENT spans are created for outgoing HTTP requests. SERVER spans are created for incoming HTTP requests. PRODUCER/CONSUMER are used for messaging systems like Kafka, SQS. INTERNAL is the default for any local operation — transformations, business logic, internal function calls.

05 — AttributesSpan Attributes and Semantic Conventions

Span attributes are key-value pairs (metadata) that can be added to spans to give more context and additional details. Spans with attributes are helpful for filtering, analysis, and alerts — they let you slice and dice your spans in your backend.

Semantic Attributes

Predefined keys that capture common metadata like http.method or db.system. They ensure standardization across systems — technical context is automatically attached by instrumentation libraries.

Custom / Business Attributes

For business context, you need to attach span attributes with manual instrumentation. Example: user.id, order.value, tenant.name. These are not automatic — you add them explicitly.

Parent-Child Timing

Parent spans usually wrap around their child spans — a parent's duration covers the time of all its children. When add_event('retry') is called without a timestamp, the SDK automatically assigns the current timestamp.

06 — ResourcesDescribing Where Telemetry Comes From

A resource is a collection of attributes that identifies the source of telemetry data. Resources are attached during SDK initialization — not per span. Once set, they appear on every span, metric, and log emitted by that provider for the lifetime of the process.

🏷️
Default resource attributes
The SDK automatically adds telemetry.sdk.name, telemetry.sdk.language, and telemetry.sdk.version. The default service.name is "unknown_service" — always override this, it's the most important resource attribute you'll set.
🔍
Resource Detectors
Plugins that automatically detect your environment — OS, host, process, container, Kubernetes pod name, cloud provider. Configure the relevant detectors once and you get rich context on every span for free.
⚙️
Custom attributes via env var
Set OTEL_RESOURCE_ATTRIBUTES and OTEL_SERVICE_NAME in your environment without changing any application code. In Kubernetes, inject them via the Downward API so each pod self-identifies correctly.

07 — BaggagePropagating Business Context Across Services

OpenTelemetry Baggage is a way to propagate key-value metadata alongside your trace — across every service boundary. It travels in HTTP request headers so every downstream service in the call chain can read it.

What it carries

App-level metadata propagated alongside the trace — e.g. user-id=u2, region=us-east. Baggage is transferred via W3C Baggage standard.

How it moves

Using propagate.inject(headers) to put baggage in headers. Using propagate.extract(headers) to extract key-value pairs from baggage on the receiving end.

⚠️ Security caution

Baggage is sent via headers — no integrity checks. Risk of sensitive data exposure when propagating beyond trusted boundaries. Caution required.

W3C Baggage standard

Baggage is transmitted using the W3C Baggage specification — the same open standard as W3C TraceContext. Both travel together in request headers.

08 — Context PropagationHow a Trace Crosses Service Boundaries

Context propagation is the mechanism that turns a collection of individual spans into a connected trace. Without it, each service would start a fresh, unconnected trace — and you'd have no way to reconstruct the full request journey.

W3C TraceContext defines how trace data moves across services. It is the default standard used by OpenTelemetry.

Context Propagation — inject → HTTP headers → extract → trace continues
📨
traceparent header
Carries the core identity: <version>-<traceID>-<spanID>-<flags>. Example: 00-a3f4b2c1...-a1b2c3d4-01. The receiving service reads this, extracts the TraceId and parent SpanId, and creates a new child span that continues the same trace.
🏪
tracestate header
Optional vendor-specific data traveling alongside the trace. Carries extra tracing information as key-value pairs specific to vendors or systems involved.
🧳
baggage header
Application-level metadata like user ID, region, tenant. Example: baggage: user-id=u2, region=us-east. Propagated alongside the trace to every downstream service — not just the next hop.

The Propagator API

TextMapPropagator

The core interface for reading and writing context into carriers like HTTP headers. Compatible with HTTP, gRPC, and custom messaging protocols.

Carrier

The medium transporting the context — HTTP headers, message headers, gRPC metadata. The carrier holds the serialized context during transit.

Setter / Getter functions

Functions that know how to write data into the carrier and read it back out. They abstract the specific format of the carrier.

Inject

Called on the outbound side. Writes the current trace context into the outgoing carrier before the request leaves the service.

Extract

Called on the inbound side. Reads context from the incoming carrier so the trace continues — not starts fresh — on the receiving end.

09 — Propagation FormatsWhich Format to Use and When

The propagation format defines the exact header names and encoding used to carry the SpanContext. OpenTelemetry supports several formats for compatibility with different ecosystems.

W3C TraceContext
Use This

The official open standard. Headers: traceparent and tracestate. Vendor-neutral and the default in OpenTelemetry. If you're starting fresh, always use this.

W3C Baggage
Use This

Companion to W3C TraceContext. Carries arbitrary key-value application metadata alongside the trace. Both W3C formats travel together.

B3 (Zipkin)
Compatibility

Originally from Zipkin, still widely used. Headers: X-B3-TraceId, X-B3-SpanId, X-B3-Sampled. Use when integrating with legacy Zipkin-based systems.

Jaeger
Legacy

Jaeger's own propagation format. Useful when migrating from OpenTracing-based Jaeger setups to OpenTelemetry.

OT Trace
Legacy

The legacy OpenTracing format. Used when migrating from OpenTracing to OpenTelemetry — keeps traces connected during the transition period.

OpenCensus Binary
Legacy

Binary metadata format used in gRPC-native OpenCensus setups. Use only when integrating with systems still running OpenCensus.

Composite Propagators

Using multiple formats at the same time

A Composite Propagator lets OpenTelemetry inject and extract context using multiple formats simultaneously. For example — using both W3C TraceContext and B3 together ensures every downstream service can continue the trace regardless of which format it understands. This is the right approach when migrating mixed environments.
