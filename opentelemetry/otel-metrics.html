Metrics in OpenTelemetry
Part 1: Fundamentals

What a metric actually is, how to think about monitoring different layers of your system, and the structural properties — monotonicity, temporality, dimensions, cardinality — that determine how metrics behave and how you should interpret them.

Thejasekhar Reddy.G  ·  April 6, 2026

01 — AnatomyWhat's Inside a Metric

A metric is not just a number. Every metric data point carries four things: a name that identifies what you're measuring, a value that is the actual measurement, a unit that tells you the scale, and dimensions — key-value attributes that add context to the data point.

Anatomy of a Metric — Name, Value, Unit, and Dimensions
🏷️
Metric Name
Identifies what you are measuring. Convention is dot-separated, describing the source and measurement — container.cpu.usage, http.requests.total, db.query.duration. A good metric name is specific enough to be unambiguous and general enough to apply across instances.
🔢
Data Point Value
The raw measurement — a number. In the example above, 0.65 means 65% CPU usage. What this number means depends on the unit and the aggregation type.
📏
Unit
Describes the scale — 1 means a ratio (0 to 1), ms means milliseconds, By means bytes. Units matter for correct interpretation and aggregation. Mismatched units in your dashboards are a common source of confusion.
📐
Dimensions (Attributes)
Key-value pairs that add context to the data point. They are what makes a metric useful for filtering and debugging — without dimensions, you know CPU is at 65% but not which container, which region, or which environment. Each unique combination of dimension values creates a separate time series.

03 — Structural PropertiesMonotonicity, Temporality, Dimensions, Cardinality

Every metric has four structural properties. These properties determine how the metric should be collected, aggregated, stored, and interpreted. Getting them wrong leads to double-counting, data skew, or queries that return incorrect results.

Monotonicity and Temporality — two structural properties of every metric

Monotonicity

Monotonicity refers to the property of a metric where its value only increases (monotonic) over time, or it can decrease as well (non-monotonic). Monotonicity helps you choose the right instrument — Counter or Gauge — and ensures correct metric aggregation.

Monotonic — Counters

Value only increases. Example: http.request.count. Once a request is counted it can never be uncounted. Resets only happen when the process restarts.

Non-monotonic — Gauges

Value can go up and down. Example: cpu.usage, memory.usage. These represent a current state of something, not an accumulation.

Temporality

Temporality defines how a metric's value relates to time — whether it represents a total since the start (cumulative) or a change over an interval (delta). Choosing the right temporality impacts your ability to calculate correct rates, averages, and percentiles. Misinterpreting temporality — for example treating a cumulative metric as a delta — can lead to double-counting or underreporting.

Cumulative

A total since the start. Sum of all values. Always growing — unless the process restarts. Correct interpretation of data, storage-efficient. Some backends expect cumulative-style input.

Delta

A change over an interval — based on point in time. Delta metrics reduce duplication and can be more efficient for frequent scrapes or exports. Some backends might work better on delta data.

04 — Dimensions & CardinalityContext That Adds Meaning — and Cost

A dimension is an attribute associated with the metric. Attributes add context to the data points. One metric can have multiple attributes or multiple dimensions — each unique combination of dimension values becomes a separate time series in your backend.

Low Cardinality

Small set of known values. Example: http.method has GET, POST, PUT, DELETE. Efficient to store and query. Low cost in your backend.

High Cardinality

Many unique values. Example: user.id — potentially millions of unique values. Each becomes a separate time series. High cardinality impacts cost, memory, and performance.

Cardinality is core to diagnosing anomalies

Cardinality is core to diagnosing anomalies and outliners. High cardinality is essential for root cause analysis — but must be used carefully. Unique dimension values drive cardinality. OTel can enrich metrics with dimensions, additional labels, or resource attributes — but always think about the cardinality impact before adding a new dimension.

05 — AggregationCombining Raw Data Points Into Statistical Summaries

Aggregation is the process of combining raw metric data points into a statistical summary. It is how individual measurements from many sources become the numbers you see on your dashboard.

Sum

Counter / Monotonic

Total of all values. Used with Counters. In OTLP: Sum maps to Counter (monotonic cumulative) or UpDownCounter (non-monotonic). With histograms we find out latencies.

Gauge

Last sampled value

Last sampled value. Used when you need current state — CPU usage, memory, queue depth. In OTLP: Gauge maps directly to Gauge instrument.

Histogram

Distributed across buckets

Distributed across buckets. Histogram is the metric instrument that captures the duration of recorded values (e.g. durations, sizes). Helps compute percentiles, distributions, and summaries efficiently. Ideal for performance analysis.

Exponential Histogram

Compressed variant

Compressed variant for large-range data. More efficient than regular histograms for data that spans many orders of magnitude.

Histogram example

Tracking request durations in a web server

Metric: http.server.duration · Unit: milliseconds · Instrument Type: Histogram · Scenario: tracking request durations in a web server. Histograms help compute percentiles, distributions, and summaries efficiently — so you can answer questions like "what was the p99 latency in the last 5 minutes?"

06 — OTel Metrics ModelThe Data Journey and Four Foundations

The OTel Metrics Model stands on four foundations. Together they ensure that downstream systems don't misinterpret values.

OTel Metrics Model — four foundations

OTel applies unit normalization so downstream systems don't misinterpret values. AggregationTemporality defines whether a metric reports a running cumulative total or a delta change since last observation. Cumulative vs Delta — it's a transformation. A delta can be converted into cumulative or vice-versa, depending on backend needs. Dimension Enrichment — OTel can enrich metrics with dimensions, additional labels or resource attributes. Spatial reaggreation — Simplify attributes. Temporal reaggreation — rollup timeseries.

Metric Data Journey — Event Model → OTLP Stream → TimeSeries Model

Reaggregation

Before Metrics reaching backed it goes through some form of transformation.

Temporal Reaggregation

Adjusts the frequency of data. Converts high-frequency metrics into longer intervals (10s → 60s), reducing storage while preserving trends.

Spatial Reaggregation

Drops unwanted attributes to reduce dimensionality and avoid cardinality explosion. Simplifies attributes before sending to storage.

Delta to Cumulative Conversion

Transforms delta metrics into cumulative totals, shifting state management to Collectors. All reaggregation is applied at SDK or Collector to optimize cost, reliability, and resources.
